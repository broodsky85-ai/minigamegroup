# 미니게임 모음

바닐라 HTML/CSS/JS로 만든 정적 웹 미니게임 모음. 빌드 도구도 프레임워크도 없다.
각 게임은 **파일 하나에 HTML·CSS·JS가 전부 들어 있다.**

- 배포: GitHub Pages, `main` 브랜치 루트에서 바로 서비스
- 주소: https://broodsky85-ai.github.io/minigamegroup/
- 저장소에 자동 커밋 스크립트가 돌고 있다. 파일을 저장하면 알아서 `main`에 커밋된다.
  따로 커밋할 필요가 없지만, **푸시 전에는 `git status`로 커밋 안 된 변경이 없는지 확인할 것.**

## 오목 챌린지 (`omok.html`)

`10명의 AI를 꺾고 오목왕에 도전하라`. 다크 네온 보스전 콘셉트.

### 구조

탭 세 개(게임하기 / 기록실 / 공략집)에 게임 탭 안에 화면 두 개(보스 선택 / 대국).

**모든 단계 데이터는 `LEVELS` 배열 하나에 있다.** 보스 이름, 등장 대사, 승리 문구,
네온 색, 공략 팁, AI 파라미터가 전부 여기 있고, 보스 카드·대국 배너·공략집 카드가
이 배열에서 렌더링된다. 예전에 이름이 JS와 HTML 두 곳에 중복 정의돼 서로 어긋나 있었다.
**단계 관련 문구를 고칠 때는 반드시 `LEVELS`만 고칠 것.**

10명의 보스: 새싹, 수습생, 방패, 추격자, 함정술사, 맹공, 철벽, 심판자, 지배자, 오목왕.

### AI 성격

각 보스는 `persona: { defBias, forkWeight }`를 갖는다.

- `defBias` — 상대 모양을 얼마나 무겁게 볼지. 1보다 크면 방어형(방패 1.5, 철벽 1.55),
  작으면 공격형(맹공 0.55, 추격자 0.62).
- `forkWeight` — 한 수로 두 방향에 위협을 세우는 자리를 얼마나 노릴지 (함정술사 1.1).

**`CRITICAL_SCORE`(400000)를 절대 없애지 말 것.** 4목 이상의 위협은 성격과 무관하게
온전한 무게로 본다. 이 보호가 없으면 공격형 보스가 상대의 열린 4를 깎아 보고,
몇 수 앞까지만 읽는 탐색이 "맞불도 해볼 만하다"고 착각해 지는 수를 둔다.
실제로 이 보호가 없을 때 6단계 맹공이 열린 3을 60번 중 6번밖에 막지 못했다.

### 기록 저장

`localStorage`(키: `omok-challenge-v1`)에 해금 단계와 단계별 승리 기록을 남긴다.
`saveRecord()` / `getStats()` 뒤에 감춰뒀으니, 서버 랭킹을 붙이려면 **이 두 함수 안쪽만**
`fetch`로 바꾸면 된다. UI는 손대지 않아도 된다.

서버가 없으므로 `상위 2.7%` 같은 **전역 순위는 만들 수 없다.** 지금은 내 과거 기록
안에서의 순위(`내 8단계 기록 중 2위 / 14승`)만 보여준다. 지어낸 숫자를 넣지 말 것.

### 아직 안 만든 것

기획에는 있으나 미구현: 오늘의 도전, 친구 도전장, 칭호 시스템, 전역 랭킹, 광고 SDK.
아이템(수읽기·차례 되돌리기)은 광고 없이 판당 3회씩 바로 쓴다.

## 검증

오목을 고쳤으면 반드시 둘 다 돌릴 것.

```bash
cd tests
npm install        # 최초 1회 (jsdom)
npm test
```

- `omok.persona.test.js` — AI 코어를 떼어내 세 가지 판을 60번씩 두게 하고
  보스 이름과 실제 행동이 맞는지 센다. 브라우저 불필요.
- `omok.game.test.js` — jsdom으로 `omok.html`을 띄우고 클릭으로 게임을 둔다.
  실제로 1단계를 이겨서 승리 연출·해금·기록까지 확인한다.

jsdom에는 canvas가 없어 **공유 카드가 그려진 모습은 테스트로 확인할 수 없다.**
브라우저에서 직접 볼 것.

## 주의

- 오목 보드는 나무색을 유지한다. 네온은 보드 바깥에만. 가독성이 최우선이다.
- 모바일에서 문구가 잘리지 않게 `clamp()`와 `word-break: keep-all`을 쓴다.
- 외부 CDN·라이브러리를 쓰지 않는다. 전부 파일 안에서 해결한다.
