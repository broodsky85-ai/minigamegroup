# 미니게임 모음

바닐라 HTML/CSS/JS로 만든 정적 웹 미니게임 모음. 빌드 도구도 프레임워크도 없다.
각 게임은 **파일 하나에 HTML·CSS·JS가 전부 들어 있다.**

- 배포: GitHub Pages, `main` 브랜치 루트에서 바로 서비스
- 주소: https://broodsky85-ai.github.io/minigamegroup/
- 저장소에 자동 커밋 스크립트가 돌고 있다. 파일을 저장하면 알아서 `main`에 커밋된다.
  따로 커밋할 필요가 없지만, **푸시 전에는 `git status`로 커밋 안 된 변경이 없는지 확인할 것.**

## 오목 챌린지 (`omok.html`)

`인간계를 넘어 신계에 도전하라`. 다크 네온 보스전 콘셉트.

### 구조

탭 세 개(게임하기 / 기록실 / 공략집)에 게임 탭 안에 화면 두 개(보스 선택 / 대국).

**모든 단계 데이터는 `LEVELS` 배열 하나에 있다.** 보스 이름, 등장 대사, 승리 문구,
네온 색, 공략 팁, AI 파라미터가 전부 여기 있고, 보스 카드·대국 배너·공략집 카드가
이 배열에서 렌더링된다. 예전에 이름이 JS와 HTML 두 곳에 중복 정의돼 서로 어긋나 있었다.
**단계 관련 문구를 고칠 때는 반드시 `LEVELS`만 고칠 것.**

보스는 **15명이고 두 세계로 나뉜다.** 각 단계는 `realm`으로 어느 쪽인지 표시한다.

- **인간계 (`realm: "human"`, 1~10단계)** — 새싹, 수습생, 방패, 추격자, 함정술사,
  맹공, 철벽, 심판자, 지배자, 오목왕
- **신계 (`realm: "divine"`, 11~15단계)** — 문지기, 천리안, 파괴신, 운명신, 오목신

인간계 최종보스 오목왕을 꺾으면 `realmGate` 연출과 함께 신계가 열린다. 보스 선택
화면 위쪽의 세계 전환 버튼도 이때 풀린다. 두 세계의 마지막 보스는 `final: true`다.

### AI 성격

각 보스는 `persona`를 갖는다. 인간계는 두 손잡이로 충분하다.

- `defBias` — 상대 모양을 얼마나 무겁게 볼지. 1보다 크면 방어형(방패 1.5, 철벽 1.55),
  작으면 공격형(맹공 0.55, 추격자 0.62).
- `forkWeight` — 한 수로 두 방향에 위협을 세우는 자리를 얼마나 노릴지 (함정술사 1.1).

신계는 여기에 `centerWeight`(중앙 포석을 얼마나 챙길지)가 붙고, 별도의 `divine` 블록으로
**전술 검사를 하나씩 켠다.** 성격 수치를 흔드는 대신 "무엇을 볼 줄 아는가"를 늘리는 방식이다.

| 스위치 | 켜면 하는 일 |
|---|---|
| `forkAttack` / `forkDefense` | 한 수 뒤에 생길 이중 위협을 미리 만들거나 막는다 |
| `forcingAttack` / `forcingDefense` | 상대가 받을 수밖에 없는 강제 수순을 계산한다 |
| `opening` | 초반 중앙 포석을 챙긴다 |
| `tacticalCand` | 위 검사에만 쓰는 넓은 후보 수 (일반 탐색의 `cand`와 별개) |

신계 다섯은 전부 `randomChance: 0`이다(9·10단계도 이미 0이다). 그래서 **난이도를 무작위 실수로
조절할 수 없고**, 스위치와 탐색 폭으로만 올린다:

문지기·천리안은 이중 위협과 포석만 켠 채 후보 폭만 넓히고(`tacticalCand` 12 → 14),
파괴신에서 `forcingAttack`, 운명신에서 `forcingDefense`가 더해지며 `depth`가 5로 깊어진다.
오목신은 넷을 모두 켜고 가장 넓게(`tacticalCand` 16) 본다.

**`CRITICAL_SCORE`(400000)를 절대 없애지 말 것.** 4목 이상의 위협은 성격과 무관하게
온전한 무게로 본다. 이 보호가 없으면 공격형 보스가 상대의 열린 4를 깎아 보고,
몇 수 앞까지만 읽는 탐색이 "맞불도 해볼 만하다"고 착각해 지는 수를 둔다.
실제로 이 보호가 없을 때 6단계 맹공이 열린 3을 60번 중 6번밖에 막지 못했다.

### 기록 저장

`localStorage`(키: `omok-challenge-v1`)에 해금 단계와 단계별 승리 기록을 남긴다.
`saveRecord()` / `getStats()` 뒤에 감춰뒀으니 UI는 저장 방식을 모른다.

**웹은 localStorage가 전부지만, Android 배포판에서는 여기에 계정 동기화가 얹힌다.**
네이티브가 `window.OmokAccount`(로그인·동기화·로그아웃·계정 삭제)와
`window.OmokRecords`(기록 읽기/쓰기)를 주입하면 게임이 그걸 감지해 동작을 바꾼다.

- 브릿지가 없으면(= 그냥 웹) 계정 UI 자체가 뜨지 않는다. `accountState.native`가 그 판정이다.
- 기록이 바뀌면 `OmokAccount.recordsChanged()`로 알려 Firestore에 올린다.
- 실제 로그인과 Firestore 읽기/쓰기는 전부 네이티브(`MainActivity.java`) 쪽에 있다.
  보안 규칙은 `firestore.rules`이고 **본인 uid 문서만** 허용한다.

**브릿지 호출은 반드시 존재 확인을 거칠 것.** 웹에서 열면 `window.OmokAccount`가 없어서
그냥 부르면 게임 전체가 죽는다. 이 규약은 `tests/omok.sync.test.js`가 지킨다.

한편 **전역 순위는 여전히 만들 수 없다.** Firestore에 올리는 건 내 기록의 백업이지
다른 사람과의 비교가 아니다. `상위 2.7%` 같은 문구를 넣지 말 것. 지금 보여주는 건 내 과거
기록 안에서의 순위(`내 8단계 기록 중 2위 / 14승`)뿐이다. 지어낸 숫자를 넣지 말 것.

### 아이템과 광고

아이템은 수읽기(🔍)와 차례 되돌리기(↩️) 둘이고, `ITEM_LIMIT = 1` — **판당 한 번씩만** 쓴다.

다 쓰면 웹에서는 그걸로 끝이지만, Android 배포판에서는 보상형 광고를 끝까지 본 뒤
한 번 더 충전할 수 있다. 흐름은 이렇다:

1. 개수가 0인 아이템 버튼을 누르면 `window.OmokAds.showRewarded(type)`을 부른다
2. 네이티브가 광고를 띄우고, 상태 문구는 `window.setOmokAdMessage()`로 게임 UI에 보인다
3. 광고를 끝까지 보면 네이티브가 `window.grantOmokItemReward(type)`을 불러 1회 충전한다

여기도 마찬가지로 **`window.OmokAds` 존재 확인 없이 부르지 말 것.**
이 세 함수의 이름과 규약은 `tests/android.package.test.js`가 고정한다. 이름을 바꾸면 앱이 깨진다.

## 멸망까지 30일 (`kingdom30.html`)

즉위한 왕이 되어 하루 한 장씩 오는 사건 카드에 좌/우 두 선택 중 하나로 답하며 30일을 버틴다.

### 구조

자원 넷(`food` 🌾 / `army` ⚔️ / `people` 👑 / `gold` 💰)이 각각 0~100이고 50에서 시작한다.
**어느 하나라도 0이 되거나 100이 되면 그 자리에서 끝난다.** 넘치는 쪽도 죽는 게 핵심 규칙이다
(군사 100은 쿠데타, 국고 100은 부패). 파멸 엔딩 8종 + 30일 생존 엔딩 10종, 합쳐서 18종.
사건 카드는 78장.

- 사건 데이터는 **`KINGDOM_EVENTS` 배열 하나**에 전부 있다.
  파일 안 `사건 카드 데이터 시작` / `사건 카드 데이터 끝` 주석 사이가 그 구간이다.
  카드를 추가·수정할 때는 이 구간만 건드린다.
- 엔딩 문구는 `ENDINGS`, 판정은 `survivalEnding()`에 있다.
- 선택지는 **항상 정확히 2개**여야 한다. 렌더링과 스와이프가 이를 전제로 한다.
- 각 선택은 얻는 것과 잃는 것이 함께 있어야 한다. 순이득 선택지는 게임을 망친다.

### 생존 엔딩은 수치가 아니라 선택으로 가른다

`survivalEnding()`은 **플래그를 먼저 보고**, 아무 색깔도 남기지 않은 판만 수치로 가른다.
처음엔 `army >= 70 && gold >= 60` 식으로 수치만 봤는데, 30일을 버티는 플레이는 모든 자원이
40 언저리로 수렴해서 통일·태평 엔딩이 3만 판에 한 번도 안 나왔다. 지금은 흑랑과 전쟁했는지,
도장을 넘겼는지, 신전을 왕좌 위에 뒀는지가 결말을 정한다. **위에서부터 먼저 걸리는 것이 이기니,
특수한 결말일수록 위에 둔다.**

`EPILOGUES`는 결말 뒤에 붙는 한 줄짜리 후일담이다. 그 판에서 세운 플래그 중 배열 순서대로
최대 3줄이 나온다. 같은 엔딩이라도 뒷이야기가 달라져서 도감을 다 채운 뒤에도 다시 할 이유가 된다.
**어떤 카드로만 세우고 아무 데서도 안 읽는 플래그가 생기면 여기에 한 줄 추가하는 게 정석이다.**

### 사건 연쇄

카드의 `flags`로 세운 플래그를 다른 카드가 `require` / `forbid`로 받아 이어진다.
`drawEvent()`는 조건이 맞는 `require` 카드가 있으면 75% 확률로 그쪽을 먼저 뽑는다.
안 그러면 애써 만든 연쇄가 30일 안에 거의 안 터진다.

### 시드 (친구 도전장)

`?s=숫자`로 들어오면 그 숫자를 시드로 `mulberry32`를 돌려 **같은 순서의 사건**이 나온다.
서버 없이 "같은 판으로 겨루기"를 만드는 장치다. 카드를 추가하면 기존 시드의 판 내용은
바뀐다 — 도전장 링크는 같은 버전끼리만 같은 판이다.

기록은 `localStorage`(키: `kingdom30-v1`)에 수집한 엔딩·최장 생존일·즉위 횟수만 남긴다.

### 밸런스

`tests/kingdom30.balance.test.js`가 세 가지 자동 플레이(무작위 / 균형 / 벼랑끝 회피)를
각 1만 판 돌려 판정한다. 지키는 선:

| 기준 | 이유 |
|---|---|
| 무작위 플레이 생존율 3~25% | 아무렇게나 눌러서 살아남으면 긴장이 없다 |
| 균형 플레이 생존율 50% 이상 | 잘 보고 고른 것이 보상받아야 한다 |
| 한 사망 원인이 전체의 40% 이하 | 한 자원만 터지면 나머지 셋이 장식이 된다 |
| 파멸 엔딩 8종 모두 등장 | 못 보는 결말은 없는 결말이다 |

수치를 손볼 때 **카드 한 장씩 고치지 말 것.** 자원별 평균 편향과 전체 진폭 두 손잡이로
움직이는 게 빠르다. 실제로 처음 78장 덱은 국고만 하루 평균 -1.72로 새서(30일이면 -52)
사망의 56%가 국고 0이었다. 국고 지출을 일괄 ×0.7 하니 이번엔 무작위 생존율이 43%로
너무 쉬워져서, 전체 진폭을 ×1.3 해 14.1%로 맞췄다. 수치는 ±18을 넘기지 않는다.

## 검증

게임을 고쳤으면 반드시 전부 돌릴 것.

```bash
cd tests
npm install        # 최초 1회 (jsdom)
npm test
```

- `omok.persona.test.js` — AI 코어를 떼어내 세 가지 판을 60번씩 두게 하고
  보스 이름과 실제 행동이 맞는지 센다. 신계 5명이 다 있는지와 응답시간도 여기서 본다.
  브라우저 불필요.
- `omok.game.test.js` — jsdom으로 `omok.html`을 띄우고 클릭으로 게임을 둔다.
  실제로 1단계를 이겨서 승리 연출·해금·기록까지 확인한다.
- `omok.sync.test.js` — 네이티브 브릿지가 **없을 때** 게임이 멀쩡한지 본다.
  계정 UI가 숨는지, 브릿지를 맨손으로 부르다 죽지 않는지.
- `android.package.test.js` — 웹과 앱이 주고받는 함수 이름(`OmokAds.showRewarded`,
  `grantOmokItemReward`, `setOmokAdMessage`)을 고정한다. **debug는 구글 테스트 광고 ID,
  release만 실제 ID**인 것과 WebView가 로컬 파일 접근 권한을 안 갖는 것도 여기서 지킨다.
- `kingdom30.smoke.test.js` — jsdom으로 `kingdom30.html`을 띄워 즉위부터 도감까지
  실제로 눌러본다. 같은 시드가 같은 판을 주는지도 여기서 본다.
- `kingdom30.balance.test.js` — DOM 없이 카드 데이터와 판정 함수만 떼어내 3만 판을
  돌린다. 생존율·사망 편중 외에 **죽은 플래그**(아무도 안 세우는 플래그를 기다리는
  카드 = 영원히 안 나옴)와 순이득 선택지도 여기서 잡는다.

`kingdom30` 쪽만 빠르게 보려면 `npm run test:kingdom`.

jsdom에는 canvas가 없어 **공유 카드가 그려진 모습은 테스트로 확인할 수 없다.**
브라우저에서 직접 볼 것.

## 주의

- 오목 보드는 나무색을 유지한다. 네온은 보드 바깥에만. 가독성이 최우선이다.
- 모바일에서 문구가 잘리지 않게 `clamp()`와 `word-break: keep-all`을 쓴다.
- 외부 CDN·라이브러리를 쓰지 않는다. 전부 파일 안에서 해결한다.
